14 // 31 bit pseudo-random number generator based on:
15 //
16 // Lehmer (1951)
17 // Lewis, Goodman & Miller (1969)
18 // Park & Miller (1983)
19 //
20 // implemented according to the optimisation suggested by David G. Carta
21 // in 1990 which uses 32 bit math and does not require division.
22 // Park and Miller rejected Carta's approach in 1993. Carta provided no
23 // code examples. Carta's approach produces identical results to Park
24 // and Miller's code.
25 //
26 // Copyright public domain . . . *but*:
27 //
28 // * Please leave the comments intact so inquiring minds have a chance of
29 // * understanding how this implementation works and chasing the
30 // * references to see the strengths and limitations of this particular
31 // * pseudo-random number generator.
32 //
33 // Output is a 31 bit unsigned integer. The range of values output is
34 // 1 to 2,147,483,646 and the seed must be in this range too. The
35 // output sequence repeats in a loop of this length = (2^31 - 2).
36 //
37 // The output stream has some predictable patterns. For instance, after
38 // a very low output, the next one or two outputs will be relatively low
39 // (compared to the 2 billion range) because the multiplier is only 16,807.
40 // Linear congruential generators are not suitable for cryptography or
41 // simulation work (such as Monte Carlo Method), but they are probably
42 // fine for many uses where the output is sound or vision for human
43 // perception.
44 //
45 // The particular generator implemented here:
46 //
47 // New-value = (old-value * 16807) mod 0x7FFFFFFF
48 //
49 // is probably the best studied linear congruentual PRNG. It is not the very
50 // best, but it is far from the worst.
51 //
52 // For the background on this implementation, and the Park Miller
53 // "Minimal Standard" linear congruential PRNG, please see:
54 //
55 // http://www.firstpr.com.au/dsp/rand31/
56 //
57 // Stephen K. Park and Keith W. Miller
58 // Random Number Generators: Good Ones are Hard to Find
59 // Communications of the ACM, Oct 1988, Vol 31 Number 10 1192-1201
60 //
61 // David G. Carta
62 // Two Fast Implementations of the "Minimal Standard" Random Number Generator
63 // Communications of the ACM, Jan 1990, Vol 33 Number 1 87-88
64 //
65 // George Marsaglia; Stephen J. Sullivan; Stephen K. Park, Keith W. Miller,
66 // Paul K. Stockmeyer
67 // Remarks on Choosing and Implementing Random Number Generators
68 // Communications of the ACM, Jul 1993, Vol 36 Number 7 105-110
69 //
70 // http://random.mat.sbg.ac.at has lots of material on PRNG quality.
71 //
72 //
73 // The sequence of values this PRNG should produce includes:
74 //
75 // Result Number of results after seed of 1
76 //
77 // 16807 1
78 // 282475249 2
79 // 1622650073 3
80 // 984943658 4
81 // 1144108930 5
82 // 470211272 6
83 // 101027544 7
84 // 1457850878 8
85 // 1458777923 9
86 // 2007237709 10
87 //
88 // 925166085 9998
89 // 1484786315 9999
90 // 1043618065 10000
91 // 1589873406 10001
92 // 2010798668 10002
93 //
94 // 1227283347 1000000
95 // 1808217256 2000000
96 // 1140279430 3000000
97 // 851767375 4000000
98 // 1885818104 5000000
99 //
100 // 168075678 99000000
101 // 1209575029 100000000
102 // 941596188 101000000
103 //
104 // 1207672015 2147483643
105 // 1475608308 2147483644
106 // 1407677000 2147483645
107 // 1 2147483646
108 // 16807 2147483647
109 //
110 // Carta refers to two registers p (15 bits) and q (31 bits) which
111 // together hold the 46 bit multiplication product:
112 //
113 // | | | |
114 // 4444 4444 3333 3333 3322 2222 2222 1111 1111 11
115 // 7654 3210 9876 5432 1098 7654 3210 9876 5432 1098 7654 3210
116 //
117 // q 31 qqq qqqq qqqq qqqq qqqq qqqq qqqq qqqq
118 // p 15 pp pppp pppp pppp p
119 //
120 // The maximum 46 bit result occurs
121 // when the seed is at its highest
122 // allowable value: 0x7FFFFFFE.
123 //
124 // 0x20D37FFF7CB2
125 //
126 // which splits up like this
127 //
128 // q 31 111 1111 1111 1111 0111 1100 1011 0010
129 // p 15 10 0000 1101 0011 0
130 // = 100 0001 1010 0110
131 //
132 // In hex, these maxiumum values are:
133 //
134 // q 31 7FFF7CB2 = 2^31 - (2 * 16807)
135 // p 15 41A6 = 16807 - 1
136 //
137 //
138 // The task is to combine the two partial products p and q as if they were
139 // both parts of a 46 bit number, with the final result being modulo:
140 //
141 // 0111 1111 1111 1111 1111 1111 1111 1111
142 //
143 // when we are actually only doing 32 bits at a time.
144 //
145 // Here I explain David G. Carta's trick - in a different and much simpler
146 // way than he does.
147 //
148 // We need to deal with the p bits "pp pppp pppp pppp p" shown above.
149 // These bits carry weights of bits 45 to 31 in the multiplication product
150 // of the usual Park Miller algorithm.
151 //
152 // David Carta writes that in order to calculate mod(0x7FFFFFFF) of the
153 // complete multiplication product (taking into account the total value
154 // of p and q) we should simply add the bits of p into the bit positions
155 // 14 to 0 of q and then do a mod(0x7FFFFFFF) on the result!
156 //
157 // | | | |
158 // 4444 4444 3333 3333 3322 2222 2222 1111 1111 11
159 // 7654 3210 9876 5432 1098 7654 3210 9876 5432 1098 7654 3210
160 //
161 // 31 qqq qqqq qqqq qqqq qqqq qqqq qqqq qqqq
162 // 15 + ppp pppp pppp pppp
163 // = Cxxx xxxx xxxx xxxx xxxx xxxx xxxx xxxx
164 //
165 // Highest possible value,
166 // for q, with a value for
167 // p which would allow it:
168 //
169 // 7FFFFFFF 111 1111 1111 1111 1111 1111 1111 1111
170 // + 41A5 100 0001 1010 0101
171 // = 8000041A4 1000 0000 0000 0000 0100 0001 1010 0100
172 //
173 // The result can't be larger than 2 * 0x7FFFFFFF = 0xFFFFFFFE. So when we
174 // do the modulus operation, we will have to subtract either nothing or just
175 // one 0x7FFFFFFF. With this model of addition, the subtraction only
176 // occurs very rarely.
177 //
178 // David Carta's explanation for why this produces the correct answer is too
179 // long to repeat here. Mine is easy to understand.
180 //
181 // Lets define some labels:
182 //
183 // Q = 31 bits 30 to 0.
184 // P = 15 bits 14 to 0.
185 //
186 // If we were doing 46 bit math, the multiplication product (seed * 16807)
187 // would be:
188 //
189 // Q
190 // + (P * 0x80000000)
191 //
192 // Observe that this is the same as:
193 //
194 // Q
195 // + (P * 0x7FFFFFFF)
196 // + (P * 0x00000001)
197 //
198 // However, we don't need or want a 46 bit result. We only want that result
199 // mod(0x7FFFFFFF). Therfore we can ignore the middle line above and use for
200 // our result:
201 //
202 // Q
203 // + P
204 //
205 // This is a lot snappier than using a division, as the Schrage technique
206 // requires.
207 //
208 // Methods:
209 //
210 // seedi Set seed with a 31 bit unsigned integer between 1 and
211 // 0x7FFFFFFE inclusive. Don't use 0!
212 //
213 // ranlui Provides the next pseudorandom number as a long unsigned
214 // integer (31 bits).
215 //
216 // ranf Provides the next pseudorandom number as a float between
217 // nearly 0 and nearly 1.0.

240  // nextrand()
241  //
242  // Generate next pseudo-random number.
243 
244  // Multiplier constant = 16807 = 7^5.
245  // This is 15 bits.
246  //
247  // Park and Miller in 1993 recommend
248  // 48271, which they say produces a
249  // somewhat better quality of
250  // pseudo-random results.
251  //
252  // 48271 can't be used with the
253  // following implementation of Carta's
254  // algorithm, since it is 16 bits and
255  // would result in bit 31 potentially
256  // being set in lo in the first
257  // multiplication. (A similar problem
258  // occurs later with the higher bits of
259  // hi.)
260 
261 #define consta 16807
262  // Modulus constant = 2^31 - 1 =
263  // 0x7FFFFFFF. We use this explicitly
264  // in the code, rather than define it
265  // somewhere, because this is a value
266  // which must not be changed and should
267  // always be recognised as a zero
268  // followed by 31 ones.
269 
270 
272  long unsigned int nextrand()
273  {
274  // Two 32 bit integers for holding
275  // parts of the (seed31 * consta)
276  // multiplication product which would
277  // normally need a 46 bit word.
278  //
279  // lo 31 bits 30 - 0
280  // hi 30 bits 45 - 16
281  //
282  // These overlap in their value.
283  //
284  // Bit 0 of hi has the same weight in
285  // the result as bit 16 of lo.
286 
287  long unsigned int hi, lo;
288 
289  // lo = 31 bits:
290  //
291  // low 16 bits (15-0) of seed31
292  // * 15 bit consta
293 
294  lo = consta * (seed31 & 0xFFFF);
295 
296  // hi = 30 bits:
297  //
298  // high 15 bits (30-16) of seed31
299  // * 15 bit consta
300 
301  hi = consta * (seed31 >> 16);
302 
303  // The new pseudo-random number is the
304  // 46 bit product mod(0x7FFFFFF). Our
305  // task is to calculate it with 32
306  // bit words and maths, and without
307  // division.
308  //
309  // The first section is easy to
310  // understand. We have a bunch of
311  // bits - bits 14 to 0 of hi -
312  // which overlap with and carry the
313  // same weight as bits 30 to 16 of
314  // lo.
315  //
316  // Add the low 15 bits of hi into
317  // bits 30-16 of lo.
318 
319  lo += (hi & 0x7FFF) << 16;
320 
321  // The result may set bit 31 of lo, but
322  // it will not overflow lo.
323  //
324  // So far, we got some of our total
325  // result in lo.
326  //
327  // The only other part of the result
328  // we need to deal with is bits
329  // 29 to 15 of hi.
330  //
331  // These bits carry weights of bits
332  // 45 to 31 in the value of the
333  // multiplication product of the usual
334  // Park-Miller algorithm.
335  //
336  // David Carta writes that in order
337  // to get the mod(0x7FFFFFF) of the
338  // multiplication product we should
339  // simply add these bits into the
340  // bit positions 14 to 0 of lo.
341  //
342  lo += hi >> 15; //
343  // In order to be able to get away with
344  // this, and to perform the following
345  // simple mod(0x7FFFFFFF) operation,
346  // we need to be sure that the result
347  // of the addition will not exceed:
348  //
349  // 2 * 0x7FFFFFFF = 0xFFFFFFFE
350  //
351  // This is assured as per the diagrams
352  // above.
353  // Note that in the vast majority of
354  // cases, lo will be less than
355  // 0x7FFFFFFFF.
356 
357  if (lo > 0x7FFFFFFF) lo -= 0x7FFFFFFF;
358 
359  // lo contains the new pseudo-random
360  // number. Store it to the seed31 and
361  // return it.
362 
363  return ( seed31 = (long)lo );
364  }
365 };
